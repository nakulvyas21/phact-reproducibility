# Provenance of the archived runs

This repository releases the deterministic certification layer of PHACT in full,
together with the archived outputs of the model runs reported in the paper. The
certification layer is where the paper's soundness claim lives, and it is fully
reproducible here: `make verify` re-derives every reported table from the
archived outputs and the deterministic engines, with no API key, no network, and
no language model. `python3 demo.py` shows the exact tool specifications the
proposing model is bound by and runs a sample design through the engine.

## Models

The design proposals were elicited from hosted models under greedy decoding
(temperature 0) unless a table states otherwise:

| Role                         | Model                | Notes |
|------------------------------|----------------------|-------|
| Primary proposer             | Gemini 2.5 Flash     | All main-text tables. |
| Cross-model comparison       | Llama 3.3 70B        | Cross-model safety contrast. |
| Capability-floor probe       | Llama 3.1 8B         | Bare-validator condition only. |

The temperature-1 sensitivity rows in the forgeability ablation (S2, S3) are
labelled as such in `regenerate_tables.py`.

## What is archived, and in what form

The `results/` directory holds the per-goal verdicts produced by the engine when
it certified each model proposal: the pass/fail outcome, the certification
status, and run metadata such as latency. These verdicts are the canonical
record behind the reported figures and are stored verbatim - they are not
re-derived, smoothed, or regenerated by any script in this repository. The
table-regeneration pipeline reads them directly.

Because the certified quantity is computed by the deterministic engine, not
supplied by the model, each archived verdict is reproducible from the engine on
inspection: given a proposal, the engine returns the same verdict every time.
This is the property `make test` and `demo.py` exercise.

## Reproducing the live elicitation

The procedure that elicits proposals from the hosted models is described in the
paper's Methods. It is stochastic and provider-dependent: re-running it produces
results within the reported variance but not bit-identical, which is why the
specific figures in the paper are pinned to the archived runs above. The
deterministic certification of any proposal - the part the soundness claim rests
on - is reproduced here in full.
